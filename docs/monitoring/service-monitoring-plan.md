# 后台服务监控落地方案

## 1. 目标

在现有评分平台后端中增加一个“服务监控面板”，让管理员可以在前端直接查看以下信息：

- 数据库连接与运行状态
- Redis 连接与运行状态
- 应用运行状态、JVM 内存、线程、GC
- 服务器 CPU、内存、磁盘占用
- 关键配置的只读展示（脱敏后）

目标不是把所有运维能力都塞进业务系统，而是先在后台里做一套“够用、可控、可审计”的轻量监控视图。

---

## 2. 现状判断

结合当前项目结构，适合直接落地的基础条件已经具备：

- 项目已经使用 Spring Boot、Spring Security、Redis、MyBatis。
- 管理端路由已经有明确权限边界，`/admin/**` 已被限制给管理员角色。
- 项目中已经存在数据库和 Redis 的连接配置，适合做状态展示和健康检查。
- 当前后端没有看到现成的 Actuator 依赖，因此监控能力需要新增。

这意味着最稳妥的方案是：

1. 先接入 Spring Boot Actuator，获取健康状态和基础指标。
2. 再补一组管理员专用的监控接口，统一给前端面板使用。
3. 对数据库、Redis、系统资源做脱敏和聚合展示，避免暴露敏感配置。

---

## 3. 推荐方案

### 方案 A：业务后台内置监控面板，优先推荐

适合当前项目的主方案。

#### 组成

- Spring Boot Actuator：提供健康检查、指标和应用状态。
- Micrometer：采集 JVM、线程、GC、连接池等指标。
- 自定义监控控制器：查询数据库、Redis、系统资源，并返回前端友好的结构。
- 前端监控页面：展示卡片、状态灯和配置摘要。

#### 优点

- 集成成本低，和现有后台风格一致。
- 直接复用现有管理员权限体系。
- 便于把监控信息和业务管理信息放在同一个入口里。

#### 缺点

- 主机级监控精度不如专业监控平台。
- 默认不包含历史曲线、统一告警和容量分析能力。

### 方案 B：业务后台 + 专业监控平台，长期推荐

在方案 A 基础上，再对接：

- Prometheus
- Grafana
- node exporter 或等价主机采集器

#### 优点

- CPU、内存、磁盘、网络指标更完整。
- 支持历史曲线、告警和容量规划。
- 适合生产环境长期运维。

#### 缺点

- 部署复杂度更高。
- 前端后台只是展示入口，真正的采集和告警不在业务系统内完成。

### 结论

当前只做轻量级实现时，建议先落地方案 A，不把 Prometheus / Grafana 纳入本期范围。

## 4. 页面和接口设计

### 4.1 后台页面建议

监控面板可以拆成以下几个卡片区域：

- 总览卡：应用状态、启动时间、环境、版本号、实例数。
- 数据源卡：数据库连接地址、连接池状态、连接是否健康。
- Redis 卡：Redis 连接地址、数据库编号、连接是否健康、池状态。
- JVM 卡：堆内存、非堆内存、线程数、GC 统计。
- 主机卡：CPU 使用率、内存使用率、磁盘使用率。
- 配置卡：只读展示关键配置项，但必须脱敏。

### 4.2 后端接口建议

建议统一挂在管理员路由下，例如：

- `GET /admin/monitor/overview`
- `GET /admin/monitor/health`
- `GET /admin/monitor/datasource`
- `GET /admin/monitor/redis`
- `GET /admin/monitor/jvm`
- `GET /admin/monitor/os`
- `GET /admin/monitor/config`

如果想减少前端请求次数，也可以合并成一个聚合接口：

- `GET /admin/monitor/dashboard`

该接口一次返回全部信息，由前端按模块渲染。

---

## 5. 数据采集口径

### 5.1 数据库

建议返回以下字段：

- 数据库类型：MariaDB / MySQL
- 连接地址：host、port、database
- 当前连接池状态：active、idle、max、minimumIdle
- 健康状态：`UP` / `DOWN`
- 最近错误信息：仅管理员可见，且做长度限制

不建议返回：

- 完整用户名密码
- 原始 JDBC URL 中的敏感参数

### 5.2 Redis

建议返回以下字段：

- Redis 地址：host、port、database
- 连接状态：`UP` / `DOWN`
- 连接池状态：active、idle、wait
- 当前缓存策略开关状态

不建议返回：

- Redis 密码明文
- 具体 token、session、refresh 原文

### 5.3 JVM

建议返回：

- 堆内存已用 / 总量 / 最大值
- 非堆内存已用 / 总量
- 线程总数、守护线程数
- GC 次数、GC 耗时
- 启动时间、运行时长

这部分可以通过 Actuator + Micrometer 直接得到，或者在自定义接口里补充更细的聚合字段。

### 5.4 主机 CPU 和内存

建议分两种级别：

#### 轻量级实现

- 使用 JVM 或 OS 采集 API 读取当前进程所在机器的 CPU 和内存。
- 适合“先能看”的需求。

#### 标准级实现

- 预留为后续扩展，不纳入本期实现。

#### 建议结论

本期只做轻量级实现，满足后台内直接查看“当前机器大概占用情况”的需求。

## 6. 安全和权限要求

当前项目已经有管理员权限边界，因此监控模块应直接复用现有权限体系，并进一步收紧到仅 `super_admin` 可访问。

### 权限原则

- 监控接口只允许 `super_admin` 访问。
- 普通用户不可见。
- 敏感配置必须脱敏。

### 脱敏原则

- 密码字段不返回。
- Token、Cookie、Session ID 不返回原文。
- URL 只展示 host、port、database 这类无风险信息。

### 审计原则

- 监控查询建议记录审计日志。
- 重点监控接口应记录操作人、访问时间、查询范围。
- 如果以后增加“导出监控数据”，必须单独加权限控制。

---

## 7. 落地实施步骤

### 第一阶段：接入基础监控能力

1. 增加 Actuator 相关依赖。
2. 开放健康检查和基础指标。
3. 新建 `super_admin` 专用监控控制器。
4. 先实现 `overview`、`health`、`jvm` 三类接口。

### 第二阶段：补齐业务资源状态

1. 增加数据库状态读取。
2. 增加 Redis 状态读取。
3. 读取数据源连接池指标。
4. 返回脱敏后的配置摘要。

### 第三阶段：补齐主机资源展示

1. 增加 CPU、内存、磁盘指标。
2. 在前端做仪表盘和趋势图。
3. 增加定时刷新和异常状态高亮。

### 第四阶段：预留增强

1. 仅预留监控字段扩展点。
2. 保留后续接入 Prometheus / Grafana 的接口位置。
3. 本期不做告警规则和外部监控打通。

---

## 8. 推荐的后端实现边界

### Controller 层

- 只负责收集监控结果并返回。
- 不直接拼接复杂计算逻辑。

### Service 层

- 封装数据库、Redis、JVM、OS 的采集逻辑。
- 负责聚合和脱敏。

### Config 层

- 配置 Actuator 暴露项。
- 限制监控接口访问范围到 `super_admin`。

### Frontend 层

- 只展示，不参与敏感信息处理。
- 支持自动刷新和按角色显示。

---

## 9. 验收标准

满足以下条件，就可以认为这套监控方案达到可用状态：

- `super_admin` 能看到当前应用是否健康。
- `super_admin` 能看到数据库和 Redis 是否连通。
- `super_admin` 能看到 JVM 内存和线程状态。
- `super_admin` 能看到当前服务器 CPU 和内存占用。
- 页面上不出现数据库密码、Redis 密码、Token 明文。
- 监控接口仅 `super_admin` 可访问。
- 接口响应稳定，不影响主业务请求。

---

## 10. 风险点

- 如果监控接口返回过多细节，容易造成敏感信息泄露。
- 如果采集逻辑直接打到数据库或 Redis 的底层对象，可能增加请求耗时。
- 如果没有节流和缓存，监控面板高频刷新会放大系统开销。

---

## 11. 推荐最终形态

最终建议做成下面这个结构：

1. 后台内置一个“服务监控”页面。
2. 后端提供 `/admin/monitor/dashboard` 这样的聚合接口。
3. 基础指标由 Actuator 提供。
4. 数据库、Redis、JVM、OS 指标由后端聚合。
5. 本期仅保留扩展空间，不接入 Prometheus / Grafana。

这样可以兼顾“快速落地”和“后续可扩展”。
